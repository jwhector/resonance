# Design — Reference vertical slice: Creator Interview → ProfileGen

- **Status:** Approved (brainstorming) — ready for implementation planning
- **Date:** 2026-06-17
- **Implements:** ADR-0013 (reference vertical slice). Touches ADR-0004, 0005, 0008,
0009, 0010, 0011, 0012, 0015.
- **Branch:** `feat/creator-interview-profilegen`

## Goal

Build the **Creator Onboarding Interview → ProfileGen** flow end-to-end — the repo's
first real vertical slice — so every architectural pattern (auth, db + pgvector, the
typed AI agent/tool registry, streaming UI, Server Actions, the full test pyramid) is
proven once and future flows replicate it rather than reinvent it.

This is a **reference implementation**: correctness and pattern-fidelity matter more
than feature breadth. Each layer should read as the canonical example its recipe points
to.

## Decisions taken during brainstorming

1. **Execution / verification mode: mock-first.** There are no live credentials
  (`.env` absent). The slice is built fully wired for live Neon / AI Gateway / Voyage /
   Resend, but verified with fakes so CI stays green with zero credentials. It flips to
   live by adding `.env` and unsetting the fakes flag. (Matches the "build/unit-test
   without creds" guidance in `docs/working-with-agents.md`.)
2. **Profile shape: focused text profile (Option A).** ProfileGen produces
  `displayName`, `headline`, a multi-paragraph `bio`, topic `tags`, and a short list of
   text-only `offerings`. **One** embedding of the whole profile seeds the matching
   backbone. The `embeddings` table is deliberately generic so deepening granularity
   later is additive (see *Future evolution*).
3. **Build sequencing: layered increments along package boundaries (Approach 1).** Three
  green, committed increments: **(1) data + auth → (2) AI → (3) UI + wiring + E2E**.
   Each stops at green (typecheck + lint + test + build) and is a safe commit/review
   point, per `docs/working-with-agents.md`.

## End-to-end happy path

1. **Land + auth.** Visitor hits `/onboarding`, enters email → Better Auth magic-link →
  authenticated session. (Mock-first: Resend sits behind a port; E2E reads the token
   from a fake transport to complete sign-in.)
2. **Interview.** At `/onboarding/interview`, the **Weave rail** streams the opening
  question ("What brought you here today?"). The `creator-interview` agent (Sonnet) runs
   server-side through the shared runner, streaming. The transcript accumulates
   client-side.
3. **Generate.** When the interview wraps (user clicks "Generate my profile"), the client
  calls the `generateProfile(transcript)` Server Action (Zod-validated).
4. **ProfileGen + persist.** The `profile-gen` agent (Opus) turns the transcript into a
  structured profile and calls its `**saveProfile` tool**. The tool handler: writes the
   `creator_profiles` row → flips the user's roles to include `creator` → embeds the
   profile text via `@resonance/ai` → writes the `embeddings` row.
5. **Render.** The action returns the profile id; `/profile/[id]` RSC-renders the
  **profile panels**.
6. **Backbone proof.** A `findSimilarProfiles(vector)` query in `@resonance/db` is
  exercised by an integration test, proving the pgvector nearest-neighbor query runs
   end-to-end even with a single row in the table.

## Increment 1 — Data + Auth

### Data model

**Auth tables** — owned/generated by Better Auth's Drizzle schema (`user`, `session`,
`account`, `verification`). The `user` table *is* ADR-0013's "users". Add a `roles` field
(`Role[]` from `@resonance/core`, default `['member']`) via Better Auth's
additional-fields config, so onboarding flips a member → creator and the `Role` type is
exercised. Better Auth owns the naming of its tables (the sanctioned exception to the
snake_case-plural convention); domain tables follow the convention.

**creator_profiles** (domain table)

- `id` (uuid, pk), `userId` (fk → `user.id`)
- `displayName`, `headline`, `bio` (text)
- `tags` (jsonb — `string[]`)
- `offerings` (jsonb — `{ title, description }[]`) — *deliberately a jsonb column, not
its own table yet* (see *Future evolution*)
- `status` (`'draft' | 'ready'`), `createdAt`, `updatedAt`

**embeddings** (generic vector store — the future-proof part)

- `id` (uuid pk), `sourceType` (`'creator_profile'` now; `'offering' | 'post' | 'interest'` later), `sourceId` (uuid of the source row)
- `model` (text — which Voyage model produced it), `content` (the embedded text, for
debugging / re-embedding)
- `embedding` (`vector(1024)` — dims pinned to the Voyage model in `@resonance/ai`)
- `createdAt`; **HNSW index** (`vector_cosine_ops`); unique `(sourceType, sourceId, model)` so re-embeds upsert.

Zod row schemas are co-located and the TS types inferred from them (convention). The
schema is authored, then the migration is **generated** via `drizzle-kit` (never
hand-edited) — following the `add-db-migration` recipe.

### Two deliberate YAGNI calls

- **Offerings stay jsonb** on the profile for now. Promoting them to their own table —
giving each offering a stable id — is *step one* of the per-offering-embedding roadmap,
so it's deferred to exactly when it's needed.
- **The interview transcript is not persisted** to a table. Durable workflows are
deferred (ADR-0009) and the interview is streaming chat — the transcript lives
client-side during the chat and is passed to the `generateProfile` action at the end.
Persisting it is a clean later addition.

This keeps the slice's domain schema to exactly ADR-0013's set (`users` +
`creator_profiles` + `embeddings`).

### Auth (`@resonance/auth`)

Better Auth instance with the magic-link plugin, persisting to our Neon DB via the
Drizzle adapter (ADR-0005). Email delivery goes through a `**MailPort`** (Resend behind
an interface) so it can be faked. Exposes the server instance, a `getSession()` helper,
and the existing `SessionUser` type.

### DB test strategy (mock-first, no credentials)

Integration tests run the real Drizzle migrations against **PGlite with the `vector`
extension** — an in-process Postgres (WASM) that supports pgvector. So `findSimilarProfiles`
and the schema are tested with *real SQL and real cosine distance*, in CI, with zero
Docker and zero keys. Production uses the Neon serverless driver; the client is
structured so tests inject the PGlite-backed Drizzle instance. (PGlite's pgvector API to
be confirmed via Context7 at build time.)

## Increment 2 — AI layer (`@resonance/ai`)

Builds on the existing `registry.ts` (`defineAgent`, `AgentDefinition`, `AgentTool`,
`ModelId`). Adds:

- `**gateway.ts`** — AI Gateway client (AI SDK v6), resolves plain `"provider/model"`
strings, reads `AI_GATEWAY_API_KEY`. This is the **swap seam**: tests inject a fake
model here.
- `**runner.ts`** — the one shared runner, two entry points: a **streaming** path
(interview chat → wraps `streamText` with the agent's system/tools) and a
**structured** path (ProfileGen → generate + tool execution, validates tool inputs with
Zod, returns the validated `outputSchema`). Throws a typed `AgentError`. No hand-rolled
loops (ADR-0009).
- `**embeddings.ts`** — `embed(text)` / `embedProfile(profile)` via Voyage (`voyage-3.5`,
**1024 dims** → pins the `vector(1024)` column). A fake embedder returns a deterministic
vector for tests.
- `**agents/creator-interview/`** — `prompt.ts` (Weave persona; runs the onboarding
interview) + the agent (`anthropic/claude-sonnet-4-6`, streaming, no tools — completion
is user-driven via the "Generate my profile" button).
- `**agents/profile-gen/**` — `prompt.ts` + the agent (Opus, `outputSchema = ProfileGenSchema`, tool `saveProfile`). Per ADR-0013 the `**saveProfile` tool** is the
thing that proves the registry: its handler writes `creator_profiles` → flips the user role → embeds → writes `embeddings.`

### Context injection pattern (worth establishing)

`saveProfile` needs the authenticated `userId` plus a `db` / `embedder` handle — *not*
model-supplied. So profile-gen is built by a small factory `defineProfileGenAgent({ userId, db, embedder })` whose tool handler closes over that server context. The registry
type (`handler: (input) => Promise<…>`) stays untouched. This is the canonical pattern for
any context-dependent tool.

### Fake-runner seam

Unit tests inject AI SDK v6's `MockLanguageModelV2` via `gateway.ts` for deterministic
agent output; embeddings use the deterministic fake. (Exact v6 test utilities confirmed
via Context7 at build time.)

## Increment 3 — UI + wiring + E2E

### UI (`@resonance/ui`, via the `add-ui-component-from-figma` recipe)

Tokens are provisional (Figma quota was blocked during scaffolding — ADR-0012); build on
the provisional token set and finalize against Figma when quota allows.

- `**WeaveRail`** — `"use client"` streaming chat rail using AI SDK v6 `useChat`. Built
on owned primitives; scaffold the missing ones (`Input` / `Textarea`, `Badge` for tags,
`Card`) via shadcn and restyle with tokens.
- `**ProfileView**` — RSC-renderable panels (displayName, headline, bio, tags, offerings).

### Wiring (`apps/web` — thin shell; logic stays in packages)

- `app/api/auth/[...all]/route.ts` — Better Auth handler; `app/onboarding/page.tsx` —
magic-link sign-in.
- `app/api/interview/route.ts` — **streaming route handler** running the
`creator-interview` agent. Streaming chat goes through a route handler, not a Server
Action — the idiomatic AI SDK pairing for `useChat`, consistent with ADR-0008's
interactive surface.
- `generateProfile` **Server Action** — the *mutation* (Zod-validates the transcript,
pulls `userId` from the session, runs profile-gen via the runner, returns the profile
id). Thin wiring delegating to packages (ADR-0002).
- `app/profile/[id]/page.tsx` — RSC profile view.

## Cross-cutting

### Mock-first runtime seam → new ADR-0016

A `RESONANCE_FAKES=1` flag (dev/test only) swaps the gateway → mock model, the embedder →
deterministic, and Resend → a fake transport, and points the DB at PGlite. This is what
lets **E2E run the whole flow deterministically with zero credentials**, and it flips to
live by unsetting the flag + adding `.env`. Recorded as **ADR-0016** because it's a
cross-cutting pattern future flows copy. Add `RESONANCE_FAKES` to `.env.example` with a
comment.

### Testing pyramid (ADR-0011)

- **Unit (Vitest, co-located):** runner (with `MockLanguageModelV2`), embeddings (fake),
`saveProfile` tool handler + `findSimilarProfiles` (against PGlite), Zod schemas, auth
helpers.
- **Component (RTL):** `WeaveRail` (renders messages, handles input, shows streaming),
`ProfileView` (renders a profile).
- **E2E (Playwright, `apps/web/e2e`):** full flow — sign in via fake transport → conduct
interview (route handler backed by the mock model) → generate → profile renders. The
app runs with `RESONANCE_FAKES=1`.

### Error handling

Zod at every boundary (route messages, action transcript, tool inputs). Typed
`AgentError` (new, in `@resonance/ai`) and `ValidationError` (existing, `@resonance/core`).
Server Actions return typed `{ ok, … }` results; the chat surfaces a friendly failure.
Don't swallow errors — propagate to the boundary that reports them (convention).

### Maintenance contract (kept in the same increments, per ADR-0015 / conventions)

- **Architecture diagram** — the slice turns `db` / `auth` / `ai` skeleton → real and
activates AI Gateway, Voyage, Resend plus new data flows. Update the diagram
per-increment via the `update-architecture-diagram` recipe.
- **Package `CLAUDE.md`** (db, auth, ai, ui) updated from "stub/skeleton" → real, in
their increment.
- **Recipes** (`add-db-migration`, `add-ai-agent`, `add-ui-component-from-figma`)
exercised and refined as used.
- `**HANDOFF.md`** updated as each increment lands green.

## Future evolution (long-term goal)

**Embedding granularity roadmap** — the generic `embeddings` table is built to grow:

1. **Now:** one vector per creator profile (`sourceType = 'creator_profile'`).
2. **Next:** promote `offerings` from a jsonb column to their own table with stable ids,
  then embed each offering (`sourceType = 'offering'`) for offering-level matching.
3. **Then:** embed posts (`'post'`) and member interests (`'interest'`) as those flows
  land.
4. **Goal:** multi-vector "resonance" matching with metadata filters (tags / role /
  region) layered on similarity, once there's a real corpus to match against (ADR-0010).

Each step is **additive** — inserting rows of a new `sourceType` — not a schema rework.

Related deferred work: persist the interview transcript (a table) for resumability;
adopt a durable workflow layer when the ADR-0009 triggers appear (combined storefront
generation, bulk re-embedding after a model change).

## Non-goals (out of scope for this slice)

- Member onboarding, discovery search, the post feed (later slices — ADR-0013).
- Any media / image generation (text-only — ADR-0013; media storage deferred — ADR-0007).
- Stripe / payments (modeled, stubbed — ADR-0006).
- Durable workflows (deferred — ADR-0009).
- Finalizing design tokens against Figma (provisional tokens are acceptable here; finalize
when quota allows — ADR-0012).

## Success criteria

- A user can sign in, complete the streaming Weave interview, generate a profile, and see
it rendered — locally with `RESONANCE_FAKES=1` (no credentials) and, with `.env`, against
live services.
- The generated profile is persisted and embedded; `findSimilarProfiles` returns it.
- `db`, `auth`, `ai` are real (their `CLAUDE.md` no longer says "stub/skeleton").
- Green across `pnpm typecheck && pnpm lint && pnpm test && pnpm build`, plus the
Playwright E2E, at each increment boundary.
- The architecture diagram, ADRs (incl. new ADR-0016), and package `CLAUDE.md` files are
in sync with the landed code.
